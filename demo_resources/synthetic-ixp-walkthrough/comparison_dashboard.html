Current IXP Prompt And Suggested Prompt-Level Changes

This section uses a sanitized prompt snapshot plus the repo's local prompt recommendation notes. It shows the current prompt surface the dashboard is assuming and the prompt edits already being suggested locally.

Current Global Prompt

You are a healthcare document data extraction specialist processing faxed prior authorization requests for the target health plan. A file may include fax cover sheets, transmission headers, duplicate pages, referral forms, clinical records, or other attachments before the actual request form. First identify the first complete prior authorization request form in the file and ignore cover sheets or non-form pages. Then extract data only from that request form and its continuation pages. If no complete prior authorization request form is present anywhere in the file, do not force a match to partial lookalike content. In that case, return null for non-repeating fields and create no rows for repeating groups such as Diagnosis Codes and Service Lines. Documents may be typed or handwritten. Return null when a field is absent, ambiguous, or illegible. Do not invent defaults.

Current v5 Prompt

You are a healthcare document data extraction specialist processing faxed prior authorization requests for the target health plan. A file may include fax cover sheets, transmission headers, duplicate pages, referral forms, clinical records, or other attachments before the actual request form. First identify the first complete prior authorization request form in the file and ignore cover sheets or non-form pages. Then extract data only from that request form and its continuation pages. If no complete prior authorization request form is present anywhere in the file, do not force a match to partial lookalike content. In that case, return null for non-repeating fields and create no rows for repeating groups such as Diagnosis Codes and Service Lines. Documents may be typed or handwritten. Return null when a field is absent, ambiguous, or illegible. Do not invent defaults.

Suggested Prompt Change

Within the selected prior authorization request form, prefer evidence from the same labeled section as the target field. Do not borrow IDs, phone numbers, or dates from a different section when a same-section candidate exists. When a field requires inference, keep the inference local to the same table or labeled block.

Helps reduce section-crossing failures such as wrong IDs, contaminated contact data, and stale carried-forward dates.

highestType of Request / Urgent / Standard

Current v5 Prompt

The Type of Request determines the urgency of the prior authorization request. Return Urgent, Standard, or Retrospective based on the selected checkbox or explicit wording. Ignore unchecked boxes. If the selection is not clear, return null.

Suggested Prompt Change

Treat Urgent, Standard, and Retrospective as one mutually exclusive checkbox group. Inspect only the mark immediately adjacent to each label. If exactly one option is marked, return that label. Ignore unchecked labels and ignore request-action fields such as Initial, Extension, Changes, or Other Request Type. Return null if none or multiple are clearly marked.

The current eval recommends making the checkbox-group logic more explicit because urgency outputs are frequently missing.

highestEpisode Type

Current v5 Prompt

Return one of: IP, OP, BH - IP, or BH - OP. Use the treatment setting plus explicit behavioral health wording when available. If only inpatient/outpatient is clear and behavioral health is not stated, return IP or OP. Return null if unclear.

Suggested Prompt Change

Return exactly one of BH-IP, BH-OP, IP, or OP. First determine inpatient vs outpatient from Treatment Setting or explicit wording. Then determine BH only from explicit behavioral-health evidence on the request form, such as behavioral health, psychiatry, psych, mental health, substance use, detox, rehab, or a BH-specific request form. If BH evidence is present, return BH-IP or BH-OP. Use the canonical tokens BH-IP and BH-OP without spaces.

The repo's prompt recommendations call out repeated BH-IP vs IP mismatches that look like BH detection failures.

highestMember ID

Current v5 Prompt

Extract the member or subscriber identifier exactly as shown on the current request form. Prefer the plan member ID and do not substitute other identifiers unless the form uses that as the member ID.

Suggested Prompt Change

Extract only the identifier labeled Member ID, Subscriber ID, Member Number, or Insurance ID in the member-information block. Preserve all letters, digits, separators, and suffixes exactly as shown, including trailing member segments such as -01. Do not use provider IDs, NPIs, TINs, fax numbers, authorization numbers, or reference numbers.

Suggested because the current prompt is too generic about preserving the full identifier token, especially suffixes.

highestDiagnosis Codes

Current v5 Prompt

The Diagnosis Codes section is a repeatable list of ICD-10 diagnosis codes for the current request. Look for a clearly labeled diagnosis area such as DIAGNOSIS CODE, DIAGNOSIS CODES, ICD-10, DX, or MEDICAL SECTION. Codes may be shown in a horizontal row of boxes, a vertical list, or a small table. Extract every diagnosis code shown for the current request, one code per repeated row, in reading order from left to right and then top to bottom. Example: if the section shows M25.511, S46.011D, and M75.41, create three diagnosis-code rows with those values. If no complete prior authorization request form is present anywhere in the file, create no diagnosis-code rows.

Extract one ICD-10 diagnosis code from the diagnosis section of the current request. Diagnosis codes may appear in separate adjacent boxes under a shared DIAGNOSIS CODE header. Treat each visible code cell as a separate diagnosis-code row. Preserve the order on the page from left to right and then top to bottom. Accept examples such as M25.511, S46.011D, or M75.41. Return only the diagnosis code value for this row.

Suggested Prompt Change

Each diagnosis row must be a complete ICD-10 code from the current request form. Reject candidates that lack the leading alpha character or end in a dangling period. When a code is split across adjacent boxes or OCR tokens, merge the visible parts for the same code before returning it. Prefer the diagnosis section of the current request over history, problem lists, or attachment-only diagnoses.

Targets truncated codes, dropped leading letters, and diagnoses borrowed from unrelated clinical history.

highestService Line Start Date

Current v5 Prompt

Extract the requested service-line start date. Accept common date formats and standardize to MM/DD/YYYY. If the table clearly carries the value down from a prior row, use that repeated context; otherwise return null.

Suggested Prompt Change

Use carry-down only when the start-date cell for the current row is visibly blank and the same service-line table clearly uses merged or continued cells. Do not borrow dates from the request date, admission date, prior authorization history, previous packet pages, or unrelated tables. Preserve the year exactly as printed.

The largest table-level problem in the current recommendations is over-broad inference for service-line start dates.

mediumProvider IDs

Current v5 Prompt

This section contains the referring provider or referring physician details for the current request. Extract only the referring provider data and do not confuse it with the servicing provider or facility.

Extract the referring provider identifier when present. Prefer the provider-specific ID field and do not substitute NPI or TIN unless the form explicitly uses that as the provider ID.

Extract the treating or servicing provider identifier when present. Prefer the provider-specific ID field and do not substitute NPI or TIN unless the form explicitly uses that as the provider ID.

Extract the attending provider identifier when present. Prefer a payer- or provider-specific ID field and do not substitute NPI or TIN unless the form explicitly uses that as the provider ID.

Suggested Prompt Change

Search the matching provider block first and prefer labels like Provider ID, Attending ID, Referring ID, Servicing ID, or plan/provider number before giving up. Keep the do-not-substitute-NPI-or-TIN rule.

Recommended because provider ID fields are mostly missing, which suggests the current prompt is not locating the right labeled block confidently.

mediumContact block

Current v5 Prompt

Extract the contact or preparer name for the current request form.

Extract the contact phone number for the current request form.

Extract the contact fax number for the current request form.

Suggested Prompt Change

Contact Name must contain alphabetic name tokens and cannot be only a phone or fax number. Contact Phone and Contact Fax must come from the same contact/preparer block as Contact Name when such a block exists. Do not swap with provider phone/fax numbers.

The existing recommendations flag contamination between contact fields and nearby provider phone/fax fields.

mediumLength of Stay

Current v5 Prompt

Extract the requested length of stay, total requested days, or equivalent duration for the current request. Return the value verbatim if clearly stated; otherwise return null.

Suggested Prompt Change

Return the single requested LOS day count only when one value is explicitly stated. Do not convert a range such as 3-5 days or 30-42 into one endpoint unless the form explicitly marks the requested total. If only a range is present, return the range verbatim or null according to the downstream contract.

Recommended because current outputs often return loose ranges rather than a single normalized requested LOS.
